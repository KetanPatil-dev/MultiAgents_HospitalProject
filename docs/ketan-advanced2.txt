 1. SUMMARY OF WHAT THIS BRANCH CONTAINS
================================================================================

This branch consolidates the algorithmic and engineering work done on the
ketan-fastpath branch into a submission-ready state. It is built on top of
the existing fast-path code and adds five layers of improvement:

    A. Algorithm consolidation         — removed dead/redundant code paths
    B. Architectural refactor (Fix B)  — CBS-first pipeline (5.4× speedup)
    C. Bug fixes                       — three real defects identified + fixed
    D. New algorithmic ideas           — anytime CBS + TaskSwap negotiation log
    E. Pedagogical assets              — a small demo level for the video

The active algorithm set is now exactly:
    A* (foundation)  +  Hungarian (Kuhn-Munkres)  +  CBS  +  HCA*  +  Joint A*

IW has been dropped — it was only called from agent.replan(), which is
disabled in the main loop, so it was dead code. Removing it leaves a cleaner
set of three multi-agent coordination strategies on top of two foundations.

 3. OVERALL ARCHITECTURE
================================================================================

The client is a layered system. Each layer has a single responsibility and a
narrow interface to the layer above.

    +-------------------------------------------------------------+
    | LAYER 5 — Client loop  (client.py)                          |
    |   Parses the level, instantiates Manager, drains stdin/out  |
    |   for 19,500 turns. Dumps negotiations.json on exit.        |
    +-------------------------------------------------------------+
    | LAYER 4 — Manager  (manager.py)                             |
    |   Setup pipeline + per-turn BDI loop + conflict resolution. |
    |   Owns: agents[], color_tasks[], chokepoint_goals,          |
    |   box_chokepoint_blockers, negotiations[].                  |
    +-------------------------------------------------------------+
    | LAYER 3 — Coordination planners                             |
    |   CBS (primary)  /  HCA* two-pass (fallback)  /  Joint A*   |
    |   Each takes the joint State and returns per-agent plans.   |
    +-------------------------------------------------------------+
    | LAYER 2 — Single-agent planner                              |
    |   A* (space-time when constraints non-empty)                |
    |   Heuristic: DistanceMap BFS-from-goal-cell                 |
    +-------------------------------------------------------------+
    | LAYER 1 — State, Action, Color, LevelProfile                |
    |   Pure data + applicability + conflict-detection primitives.|
    +-------------------------------------------------------------+

    Setup pipeline (executed once after parsing):
        1. Build DistanceMap from box-goal cells
        2. Build per-color task queues (box-goal + agent-goal tasks)
        3. Size-gate: skip expensive steps for very large levels
        4. Corridor-clearing tasks for deco boxes (push out of choke points)
        5. Chokepoint analysis — TWO classes:
             (a) agent-blocking: delivering box X here disconnects agent Y
                 from agent Y's positional goal
             (b) box-blocking-box: delivering box X here disconnects box Z
                 from box Z's goal
           Both detected by BFS-with-cell-removed.
        6. Anytime CBS preplan (cap schedule 50→200→800→3000, 5 s budget)
        7. HCA* recovery for any agent CBS could not plan
        8. Joint A* fallback ONLY for ≤2 agents AND ≤450 cells
           (provably optimal in that regime, intractable elsewhere)

    Per-turn loop:
        1. Joint-plan fast-path: if Joint A* produced a complete sequence,
           pop the next joint action.
        2. For each agent:
             a. Pop next task if needed (skipping chokepoint-deferred ones).
             b. Mark current task solved if box/agent already at goal.
             c. Preplan if no current plan:
                  • Detect obstacles via ghost-state A*
                  • Same-color blocker → swap with own task
                  • Cross-color blocker → record TaskSwap, request foreign agent
                  • Agent blocker     → mark awaiting, nudge it
                  • Otherwise run real-state A* with runtime constraints
             d. Append next action from plan (or NoOp).
        3. Validate joint action; agents in conflict become NoOp, plans cleared,
           and a (r, c, 0) constraint is injected to force a different route on
           the next replan.
        4. Stuck-agent recovery: agents with no plan for 15 turns get a
           random valid Move from the seeded RNG (random.seed(50) → reproducible).
        5. Deadlock detection: rolling 30-step window of joint-state hashes.
           If the same hash appears ≥4 times → trigger escape cells for the
           stuck agents (BFS outward ≥6 steps, biased by agent_id).
        6. Periodic constraint pruning every 50 turns.


================================================================================
 4. PROS — what the design does well
================================================================================

    a) Three coordination strategies, each with a CLEAR niche
       - CBS for medium-scale multi-agent conflict-free joint planning
       - HCA* for cases where CBS cannot plan an agent (excluded by individual
         A* infeasibility) — priority-order constraint-registration
       - Joint A* for tiny levels where optimality is tractable
       The pipeline runs them in cheapest-first order so the easy levels
       short-circuit before the expensive ones are tried.

    b) CBS-first ordering is an empirically validated decision, not a default
       On WardRush this single change reduced setup time from ~19 s to ~1.3 s
       (5.4× total speedup, identical action count). The number is reproducible
       on demand — see "Numbers to cite" below.

    c) Cross-agent task swaps as IMPLICIT NEGOTIATION
       When agent A's plan is blocked by a box of agent B's colour, A does not
       just wait — it requests B reassign B's task to clear the obstacle. Each
       request is recorded as a TaskSwap and cycle-detected: if (requester,
       granter, obstacle_box) repeats ≥3 times in 30 turns, the chain is in
       oscillation and further swaps are refused. This replaces the original
       box_swap_count >= 5 band-aid with a principled cycle-detection mechanism.

    d) Chokepoint reasoning is two-tiered, not single-purpose
       - Agent-blocking chokepoints defer box deliveries that would strand an
         agent away from its positional goal.
       - Box-blocking-box chokepoints defer deliveries that would disconnect
         another pending box's path. The obstacle-clear helper also avoids
         pushing the obstacle INTO such cells.

    e) Optimal Hungarian, not greedy
       The original code called itself "Hungarian" but did greedy min-cost
       matching per character. This branch implements the actual O(n³)
       Kuhn-Munkres assignment algorithm. Same interface, different result on
       levels with multiple boxes/goals of the same character.

    f) Anytime CBS for the 3-minute competition budget
       Standard CBS is binary: succeed under the node cap or fail. Anytime CBS
       starts with a cheap cap (50 nodes) and escalates (→200 →800 →3000) as
       wall-clock budget allows. Easy levels return in microseconds; hard
       levels still get a chance.

    g) Reproducibility
       random.seed(50) is set once at client.py entry. The only stochastic
       paths are the obstacle-unblock nudge and the stuck-agent escape nudge,
       both of which feed from the seeded RNG. Two runs of the same level
       produce identical action sequences end to end.

    h) Architectural honesty
       The Agent class supports a full BDI lifecycle (sense / plan_is_sound /
       replan / next_action). The Manager currently bypasses it for direct
       preplanning, but the structure is preserved. This is intentional — it
       documents the original design before the runtime layer was bolted on.

 5. WHAT TO INFER / INCLUDE IN THE FINAL VIDEO
================================================================================

    The graded artifact is the video (June 1). The competition outcome does not
    directly affect the grade (per §1.4 of the assignment). The video must run
    ≤12 minutes and should follow guide_for_video.pdf.

    Frame the project around THREE big algorithmic ideas. Each is already in
    the code. Do NOT add a fourth — it dilutes the narrative.

       BIG IDEA 1 — Negotiated multi-agent coordination via task swaps
       ----------------------------------------------------------------
       "We model obstacle resolution as an implicit negotiation between
        agents. When an agent's path is blocked by a box of another agent's
        colour, it requests that agent reassign their task to clear the
        obstacle. Every request is recorded in a structured log, and a
        (requester, granter, box) tuple repeating three times in a 30-turn
        window signals oscillation that we break by refusing further swaps."

       Demo evidence:
         - On MADS: 3 negotiations logged in setup; 4th attempt refused with
           "Negotiation cycle detected: Agent 1 -> Agent 2 over box C (3x);
           refusing further swaps." — a principled break, not a hack.
         - On WardRush: 2 negotiations recorded (Agent 2 → Agent 1 over box E).
         - The TaskSwap log is dumped to negotiations.json on every run,
           enabling post-hoc visualisation.

       BIG IDEA 2 — Chokepoint-aware task scheduling
       ----------------------------------------------------------------
       "Before planning, we identify which box-goal cells are critical to
        the level's topology — cells whose closure would disconnect either
        an agent from its positional goal, or another box from its goal.
        We then defer those deliveries until the dependent task completes.
        The same analysis informs obstacle-clearing: we never push an
        obstacle into a critical cell while its blockee is still pending."

       Demo evidence:
         - PedagogicalDemo.lvl deliberately constructs this scenario.
           Setup log: "Box-chokepoint: delivering to (1,7) blocks box B
           delivery. Will defer." — and it does, producing an 18-action solve.

       BIG IDEA 3 — Hybrid pipeline with anytime CBS, HCA* fallback, Joint A*
       ----------------------------------------------------------------
       "We evaluated CBS-first vs HCA*-first orderings empirically. On
        WardRush, CBS-first reduces median setup time by 14× with identical
        plan quality, because HCA*'s rigid priority order creates artificial
        infeasibility on configurations CBS resolves trivially. We also
        wrap CBS with a wall-clock budget that escalates node-cap from 50
        to 3000 — this is a classic anytime algorithm that handles the
        3-minute competition timeout gracefully."

       Demo evidence:
         - Before Fix B: WardRush setup time ~19 s (HCA* fails on 3 agents,
           17 s wasted); total 21.6 s.
         - After Fix B: WardRush setup ~1.3 s, total 4.0 s, same 267 actions.
         - 5.4× total speedup is a concrete number you can defend.


    Numbers to cite verbatim:

       - Ccropper:         25 actions, 0.02 s
       - crayCray:         91 actions, 3.2 s

 6. WHAT'S GOOD ABOUT THE MODEL / AI ITSELF
================================================================================

    Not just "the code" — what specifically demonstrates AI/algorithmic depth.

    a) FOUR DISTINCT ALGORITHMS that are not just textbook copies:
         - Kuhn-Munkres Hungarian (textbook, but implemented from scratch)
         - CBS with constraint trees (textbook, but with anytime wrapper)
         - HCA* with two-pass projected-state retry + box-trajectory time
           constraints (this two-pass form is uncommon)
         - Joint A* (textbook, but with carefully-tuned gating)

    b) ORIGINAL ALGORITHMIC IDEAS:
         - Task swap as implicit negotiation (the TaskSwap log is the actual
           mechanism, not a wrapper around something else)
         - Box-blocking-box chokepoint analysis (the agent-blocking case is
           occasionally seen in literature; the box-blocking variant via BFS-
           with-cell-removed is original)
         - Obstacle-clear avoidance of pending chokepoints — the
           _find_obs_clear_goal helper actively refuses to push obstacles into
           cells that would create a new chokepoint

    c) ENGINEERING JUDGMENT visible in the code:
         - Size-based gating (big_level / very_big_level flags) skips expensive
           preprocessing on levels where it would burn the 3-min budget
         - Anytime CBS escalates rather than committing to a single node cap
         - Pipeline orderings are cheap-first so easy levels short-circuit
         - The deadlock detector has a 15-step cooldown to avoid thrashing

    d) DEFENSIVE BUG-FIXING — three real defects identified and fixed in
       this branch:
         - Three print() calls writing to stdout instead of stderr — the
           stdout channel is the protocol bus to the server, and any leaked
           print causes "Invalid number of agents in joint action" errors.
         - _chokepoint_blockers_pending() checked only agent-blocking
           chokepoints, never box-blocking ones (the box_chokepoint_blockers
           dict was populated but never consulted).
         - _find_obs_clear_goal() could push the obstacle INTO a chokepoint
           cell, defeating the defer that had just been set up.

    e) REPRODUCIBLE BEHAVIOUR — random.seed(50) at client.py top, plus the
       few remaining stochastic decisions all draw from the same seeded RNG,
       so two runs of the same level give identical action sequences. This
       matters because graders may re-run your client; non-determinism makes
       results impossible to trust.

    f) STRUCTURED OUTPUT — negotiations.json is written at the end of every
       run with the full TaskSwap log (requester, granter, obstacle_box,
       obstacle_pos, original_task, new_task, t_initiated, same_color). This
       turns "internal state" into "data the team can analyse and visualise
       offline" — the bridge to the video's annotated playback.

    g) READABLE CODEBASE — dead code, commented-out blocks, alarming
       `# this is a bug` comments, manager.py.backup, the random-move band-
       aid, and the box_swap_count counter have all been removed. The
       remaining code reflects the actual decisions, not the history of
       trying things.

 7. RUNNING THE CLIENT
================================================================================

    Foundation deps (one-time):
        - Python ≥3.10 (uses union-type syntax)
        - psutil >=6.1.1
        - Java for the MAvis server (server.jar shipped with the repo)

    Run a single level with the GUI:
        ./test.sh WardRush.lvl

    Run headless with a 3-min timeout (competition profile):
        PYTHONPATH=searchclient_python \
            java -jar server.jar \
                -l WardRush.lvl \
                -c "python3 -m searchclient.client --bypass-debugger" \
                -t 180 \
                -o WardRush_run.zip

    Output:
        - WardRush_run.zip                  server log (replayable with -r)
        - searchclient/negotiations.json    structured TaskSwap log for the run


================================================================================
 8. WHAT IS STILL OPEN
================================================================================

    These are known and out of scope for the May 21 code submission:

    a) Five previous-years levels currently fail (MAybetwo, TeamAgent, noone,
       MADS, DECrunchy, DatzCrazy) due to level-specific deadlock patterns
       that the current obstacle-swap layer does not unwind. Each is days of
       debugging; fixing one rarely generalises. Coverage on the May 18
       competition levels is the true measure to optimise against.

    b) The pedagogical demo level fires chokepoint cleanly. We may want a
       second small demo level designed specifically to fire negotiation
       (the current PedagogicalDemo's geometry causes chokepoint to handle
       the situation before negotiation triggers).

    c) The deadlock detector and stuck-agent recovery currently re-trigger
       with a 15-step cooldown. On levels with very long plan horizons this
       may be too aggressive; consider making the cooldown adaptive to plan
       length.

    d) Joint A* heuristic (sum of per-agent BFS dists to goal) is not
       admissible in cases where two agents' paths interfere. This is a
       known issue; the gating to ≤2 agents avoids it in practice. Could
       switch to max-of-per-agent-cost if extending Joint A* upwards.
